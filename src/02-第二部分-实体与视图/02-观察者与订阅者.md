# 第 5 章：观察者与订阅者

有了 Entity 存储数据，下一步是**响应数据变化**。GPUI 提供了三种通信原语：`observe`、`subscribe`、和 `on_action`（后者在第 15 章讨论）。

> 源文件参考：`src/app.rs` 的 `observe`/`subscribe`/`emit`/`notify` 方法、`src/app/context.rs`

## 两种通信范式

### observe：通知"变了"（不含数据）

```rust
cx.observe(&other_entity, |observer: &mut Self, observed: &OtherView, cx| {
    // OtherView 发出了 notify——但我不知道为什么
    // 我需要自己检查是什么变了
})
```

- 触发条件：`cx.notify()` 被调用。
- 携带信息：零。你只被告知"它变了"。
- 使用场景：观察一个实体的整体状态变化，然后读取最新值。

### subscribe：通知"发生了什么"（含数据）

```rust
cx.subscribe(&other_entity, |observer: &mut Self, event: &SomeEvent, cx| {
    // 携带具体事件数据
    match event {
        SomeEvent::Incremented { by } => { /* ... */ }
        SomeEvent::Reset => { /* ... */ }
    }
})
```

- 触发条件：`cx.emit(SomeEvent::Incremented { by: 5 })`。
- 携带信息：类型化的事件对象。
- 使用场景：精确通信——你知道发生了什么，不需要重新读取。

## 内部实现

### Observer 的存储

```rust
// App 中
observers: FxHashMap<EntityId, Vec<AnyObserver>>,
// AnyObserver 存储了回调函数 + observer 的 EntityId
```

当 `flush_effects()` 处理 `Notify` 效果时：

```
1. 查找 EntityId 的 observers 列表
2. 遍历每个 observer：
   a. 检查 observer 实体是否还存活
   b. 从 EntityMap 中租借出 observer 实体的 &mut Self
   c. 从 EntityMap 中租借出 observed 实体的 &Entity
   d. 调用回调
   e. 归还两个租借
```

注意：observer 回调带 `&mut Self`——你可以在 observer 中修改自己的状态。这是观察者模式的惯用写法。

### Subscriber 的存储

```rust
// App 中
event_listeners: FxHashMap<(EntityId, TypeId), Vec<AnyEventListener>>,
// key 是 (发射者 ID, 事件类型 ID)
```

当 `flush_effects()` 处理 `Emit` 效果时：

```
1. 查找 (EntityId, TypeId(E)) 的 subscribers 列表
2. 遍历每个 subscriber：
   a. 从 EntityMap 中租借出 subscriber 的 &mut Self
   b. 传入 &E（事件对象的引用）
   c. 调用回调
```

### Subscription 类型

`cx.observe()` 和 `cx.subscribe()` 返回 `Subscription`：

```rust
pub struct Subscription {
    // 内部持有清理所需的信息
    // Drop 时自动取消订阅
}
```

当你把 `Subscription` 存储在自己的结构体中时，它会在你的实体被 drop 时自动清理——不需要手动取消订阅。这是 GPUI 防止"野指针"观察者的关键机制：**观察者总是与被观察者同生命周期**。

## 完整示例：两种模式对比

```rust
// 被观察的实体
struct Counter {
    count: i32,
}

// EventEmitter 为 Counter 提供 emit 能力
impl EventEmitter<CounterEvent> for Counter {}

enum CounterEvent {
    Incremented { by: i32, new_value: i32 },
    Reset,
}

// 观察者实体
struct Display {
    counter: Entity<Counter>,
    _subscription: Subscription, // 持有以维持订阅
}

impl Display {
    fn new(counter: Entity<Counter>, cx: &mut Context<Self>) -> Self {
        let subscription = cx.subscribe(
            &counter,
            |this: &mut Self, event: &CounterEvent, cx| {
                // 接收类型化事件——知道具体发生了什么
                match event {
                    CounterEvent::Incremented { by, new_value } => {
                        println!("Counter incremented by {} to {}", by, new_value);
                    }
                    CounterEvent::Reset => {
                        println!("Counter was reset");
                    }
                }
            },
        );

        Self { counter, _subscription: subscription }
    }
}

// 只需感知变化时用 observe
impl Render for Counter {
    fn render(&mut self, _window: &mut Window, cx: &mut Context<Self>) -> impl IntoElement {
        div()
            .child(format!("Count: {}", self.count))
            .on_click(cx.listener(|this, _event, cx| {
                this.count += 1;
                cx.notify(); // 触发所有 observe 此实体的回调
                cx.emit(CounterEvent::Incremented { // 触发 subscribe 此事件的回调
                    by: 1,
                    new_value: this.count,
                });
            }))
    }
}
```

## 运行时借用限制

Observer 和 subscriber 回调都要求对 observer 实体（`&mut Self`）和 observed 实体（`&T` 或 `&E`）进行租借。这意味着：

1. **不能在一个 observer 中再次 notify 同一个实体**——这会导致递归和潜在的无限循环。GPUI 在开发模式下会 panic。
2. **不能同时观察一个实体并获取其 `&mut T`**——因为 observer 回调已经借出了观察者的 `&mut Self`。

这些限制在实践中通常不明显，但如果你遇到 `BorrowError` panic，通常意味着你在 effects 处理链中造成了循环。

## 最佳实践

1. **emit 用于"发生了什么"，notify 用于"我不确定变没变"**：emit 携带精确信息，让 subscriber 做出精确响应。notify 更粗糙，但实现成本更低。
2. **WeakEntity 避免循环**：如果 A 观察 B，B 的字段中不要持有 `Entity<A>`——改用 `WeakEntity<A>`。
3. **Subscription 必须持有**：如果 `cx.observe()` 的结果被立即 drop，订阅会立刻取消。这是常见的"订阅不生效" bug。

---

**如果你只记住一件事：** `notify()` + `observe()` = 感知变化；`emit()` + `subscribe()` = 接收事件。前者无数据，后者有数据。选择哪个取决于"接收者是否需要知道具体发生了什么"。
