# 附录 C：术语对照

GPUI 的术语有精确含义。混用这些术语会导致沟通歧义——尤其是在提问和阅读源码时。

## Entity vs View vs Component

| 术语 | 定义 | 代码标识 | 是否实现 Element |
|------|------|---------|-----------------|
| **Entity** | 存储在 EntityMap 中的任意数据，通过 `Entity<T>` 句柄访问 | `cx.new(\|_\| T { ... })` | 否 |
| **View** | 实现了 `Render` 的 Entity，可以产生 UI | `impl Render for T` | 是（通过 `Entity<V: Render>` 实现） |
| **Component** | 实现 `RenderOnce` 的类型，每帧创建/销毁 | `#[derive(IntoElement)]` + `impl RenderOnce` | 是 |

准确的说法：
- "这个 view 的 render() 返回一个 div" ✅
- "这个 entity 存了应用配置" ✅
- "这个 component 是一个 button" ✅
- "这个 entity 是一个 element" ❌（entity 不直接是 element）

## Element vs Drawable vs Node

| 术语 | 定义 | 生命周期 |
|------|------|---------|
| **Element** | 实现 `Element` trait 的类型——定义了如何布局和绘制 | trait |
| **Drawable** | 框架内部的包装类型，管理 Element 的状态机 | 单帧 |
| **AnyElement** | 类型擦除的、堆分配的 Element | 单帧 |
| **Node** | Taffy 布局引擎中的节点 | 单帧（布局计算期间） |

## Style vs StyleRefinement

| 术语 | 作用 |
|------|------|
| **Style** | 完整的具体样式——所有字段都有值 |
| **StyleRefinement** | 样式的"增量修改"——每个字段是 `Option<T>` |
| **refine()** | 将 StyleRefinement 覆盖到 Style 上（`None` 不覆盖） |

类比 CSS：
- `Style` = computed styles（每个属性都有最终值）
- `StyleRefinement` = inline style 或单个 CSS 规则（只设置了部分属性）

## Observer vs Subscriber

| 术语 | 触发方式 | 携带数据 |
|------|---------|---------|
| **observe** | `cx.notify()` | 无（需要自己读取最新值） |
| **subscribe** | `cx.emit(E)` | 有（类型化事件对象） |

## App vs AsyncApp vs Context

| 术语 | struct | Send | 生命周期 | 使用位置 |
|------|--------|------|---------|---------|
| **App** | `App` | ❌ | 主函数/update 回调 | `main()` |
| **AsyncApp** | `async_context::AsyncApp` | ✅ | `'static` | `cx.spawn()` |
| **AsyncWindowContext** | `async_context::AsyncWindowContext` | ✅ | `'static` | `cx.spawn_in()` |
| **Context** | `Context<'a, T>` | ❌ | `'a` | `render()`, 事件处理 |
| **TestApp** | `TestAppContext` | ✅ | `'static` | `#[gpui::test]` |

## WindowHandle vs AnyWindowHandle

| 术语 | 类型 | Copy |
|------|------|------|
| **WindowHandle\<V\>** | 知道根视图类型 V | ✅ |
| **AnyWindowHandle** | 类型擦除 | ✅ |

## FocusHandle vs WeakFocusHandle

| 术语 | 内部类型 | 是否阻止焦点 entry 释放 |
|------|---------|----------------------|
| **FocusHandle** | `Arc<FocusState>` | 是 |
| **WeakFocusHandle** | `Weak<FocusState>` | 否 |

## Entity/AnyEntity/WeakEntity

| 术语 | 类型 | 阻止释放 |
|------|------|---------|
| **Entity\<T\>** | 保留 T | 是 |
| **WeakEntity\<T\>** | 保留 T | 否 |
| **AnyEntity** | 类型擦除 | 是 |

## Paint vs Render vs Draw

这三个词在 GPUI 中有明确区分：

| 术语 | 含义 |
|------|------|
| **render()** | View 的方法——从数据生成元素树 |
| **paint()** | Element 的方法——从元素生成 Scene 图元 |
| **draw()** | Window 的方法——触发整帧的渲染周期 |

时序关系：
```
draw() 调用:
  1. View::render() → 元素树
  2. Element::paint() → Scene
  3. GPU render Scene → 屏幕
```

## Scene vs DisplayList

在 GPUI 0.2.2 中，`Scene` 就是一个 display list（绘制命令列表）。它**不是**场景图（scene graph）——没有树形结构，没有父子节点的 transform 累积。历史名字沿用至今。

## Platform vs Backend

| 术语 | 含义 |
|------|------|
| **Platform** | 操作系统 + 窗口系统（macOS, Windows, Wayland, X11） |
| **Backend** | GPU 渲染后端（Metal, DirectX 11, Blade） |

一个 Platform 选择一个 Backend：macOS 默认用 Metal，Linux 用 Blade，Windows 用 DirectX 11。

## Slotmap / SecondaryMap / EntityMap

| 术语 | 是什么 |
|------|--------|
| **SlotMap** | 稀疏映射（key 是整数，支持 O(1) 插入/删除/查找，key 可回收） |
| **SecondaryMap** | slotmap 的关联数据结构——与 SlotMap 共享同一个 key 空间 |
| **EntityMap** | GPUI 对 SecondaryMap 的封装，存储所有实体 |

## Arena

GPUI 每帧使用一个 `Arena` 分配器来分配 `Drawable` 树中的节点。帧结束时空整个 Arena 被清空——这是一种高性能的批量释放策略（比逐个 `drop` 每个 `Box<dyn Element>` 快得多）。

## 命名约定速查

| 前缀/后缀 | 含义 | 示例 |
|----------|------|------|
| `Any*` | 类型擦除 | `AnyView`, `AnyElement`, `AnyEntity` |
| `Weak*` | 弱引用 | `WeakEntity`, `WeakFocusHandle` |
| `*Refinement` | 增量修改（Option 字段） | `StyleRefinement`, `TextStyleRefinement` |
| `*Handle` | 轻量句柄 | `WindowHandle`, `FocusHandle` |
| `*Id` | 整数标识 | `EntityId`, `ElementId`, `LayoutId`, `FocusId` |
| `Platform*` | 平台抽象 | `PlatformWindow`, `PlatformDisplay` |
