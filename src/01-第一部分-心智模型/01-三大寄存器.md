# 第 1 章：三大寄存器

GPUI 需要存储状态。你的应用有数据——计数器是 3 还是 5，下拉菜单是展开还是收起，滚动位置在第几行。这些状态必须放在某个地方。

GPUI 提供了三个"寄存器"（register），对应三种不同生命周期和用途的状态存储。理解这三个寄存器的区别，是你理解 GPUI 的第一块基石。

> 源文件参考：`src/app/entity_map.rs`、`src/window.rs`、`src/global.rs`

## 寄存器一：EntityMap（实体存储）

```
App
  └── EntityMap
        ├── EntityId(0) → Box<dyn Any> → Counter { count: 3 }
        ├── EntityId(1) → Box<dyn Any> → TodoList { items: [...] }
        └── EntityId(2) → Box<dyn Any> → Settings { theme: Dark }
```

`EntityMap` 是 App 拥有的一个 slot map（槽位映射）。你创建的所有"模型"——即带 `Render` 或 `EventEmitter` 的数据结构——都以 `Box<dyn Any>` 的形式存储在这个 map 里。

你手里的 `Entity<T>` 不是一个指针，它不直接指向数据。它是两样东西的组合：

1. **`EntityId`**：一个整数 key，标记数据在 slot map 中的位置。
2. **引用计数**：`Arc<Weak<RwLock<EntityRefCounts>>>`，用于判断实体是否还在被使用。

这意味着：

```rust
let handle_a: Entity<Counter> = cx.new(|_| Counter { count: 0 });
let handle_b: Entity<Counter> = handle_a.clone();
// handle_a 和 handle_b 共享同一个 EntityId，指向 EntityMap 中的同一条数据。
// 只有所有 Entity<Counter> 和 WeakEntity<Counter> 都释放后，数据才从 map 中移除。
```

**什么时候用 Entity？**

- 多组件共享可变状态。
- 状态需要在事件处理和渲染之间持久化。
- 需要用 `cx.notify()` / `cx.emit()` 触发重绘。

## 寄存器二：Element State（元素状态）

```
Window
  └── Frame.element_states: FxHashMap<(GlobalElementId, TypeId), ElementStateBox>
        ├── ([root, scroll_area, scrollbar], TypeId(ScrollState)) → ScrollState { offset: 150.0 }
        ├── ([root, panel, toggle], TypeId(ToggleState)) → ToggleState { expanded: true }
        └── ([root, editor, line_42], TypeId(EditorState)) → EditorState { ... }
```

这是每个 `Window` 的 `Frame` 内维护的一个 HashMap。key 是 `(GlobalElementId, TypeId)` 二元组——`GlobalElementId` 是一个 `SmallVec<[ElementId; 32]>`，记录了从根元素到当前元素的 ID 路径；`TypeId` 区分同一元素上的不同类型状态。value 是 `ElementStateBox { inner: Box<dyn Any> }`。

状态在双缓冲的两个 `Frame`（`rendered_frame` 和 `next_frame`）之间迁移：每帧结束时，上一帧中仍被访问的 state 迁移到新帧，未被访问的自动丢弃。

关键规则：**状态在跨帧时持久化，当且仅当 GlobalElementId 保持不变**。

如果你的元素树结构中某个祖先元素的 ID 变了，或者元素在条件渲染中消失了——整条路径失效，状态丢失。

```rust
// 危险：条件渲染导致 ID 路径变化
if expanded {
    div().id("outer").child(
        div().id("inner").child(/* 这个元素的状态在 expand 切换时会丢失 */)
    )
}

// 安全：始终渲染，用 visibility 或 overflow 隐藏
div().id("outer").child(
    div().id("inner").when(expanded, |d| d.visible(true))
)
```

**什么时候用 Element State？**

- 纯 UI 状态：滚动位置、展开/折叠、拖拽位置、动画进度。
- 状态不需要被其他组件访问——它是"这个元素私有的"。
- 通过 `Stateful<E>` 或直接实现 `Element` trait 来存储。

## 寄存器三：Globals（全局状态）

```
App
  └── globals_by_type: FxHashMap<TypeId, Box<dyn Any>>
        ├── TypeId(Theme) → Theme::Dark
        ├── TypeId(FontCache) → FontCache { ... }
        └── TypeId(UserPreferences) → UserPreferences { ... }
```

`App` 内部有一个 `HashMap<TypeId, Box<dyn Any>>`，每个类型最多存一个实例。这是 GPUI 版的"单例"。

```rust
// 设置全局
cx.set_global(Theme::Dark);

// 读取全局
let theme = cx.global::<Theme>();

// 监听全局变化
cx.observe_global::<Theme>(|cx| {
    // Theme 变了，刷新所有窗口
});
```

**什么时候用 Global？**

- 应用级配置：主题、语言、字体设置。
- 需要大量组件访问的共享数据。
- 类型本身只有一个有意义实例。

## 三者对比

| | EntityMap | Element State | Globals |
|---|---|---|---|
| **存储位置** | App.entity_map | Frame.element_states | App.globals_by_type |
| **键** | EntityId（整数） | (GlobalElementId, TypeId)（ID 路径 + 类型） | TypeId（类型哈希） |
| **生命周期** | 所有句柄释放时回收 | 元素 ID 路径变化时丢失；两帧未被访问则丢弃 | 显式设置新值时替换 |
| **触发重绘** | notify() / emit() | 自动（元素重绘） | 无自动通知（需搭配 observe_global） |
| **典型用途** | 业务状态 | UI 组合状态 | 全局配置 |

## 常见误区

**误区一：Element State 是 Entity 的"缓存"**

不是。它们是两个完全独立的存储位置，不存在自动同步。如果你把 `Entity` 里的某个字段和 element state 里的某个值各存了一份，你需要自己保证一致性。

**误区二：所有 UI 状态都应该放在 Entity 里**

Entity 开销更大（引用计数、notify 传播）。纯 UI 的、不需要跨组件共享的状态，放在 element state 更高效。

**误区三：Global 可以存多个同类型实例**

不行。如果你需要存储多个 `Settings` 实例，把它们放进一个 `HashMap<String, Settings>` 然后把这个 HashMap 作为 global。

---

**如果你只记住一件事：** 数据在 EntityMap、element state、globals 中三选一，它们之间没有自动同步——选错存储位置是状态丢失 bug 的根源。
