# 第 3.5 章：跨帧机制与双缓冲

第 2 章讲了帧循环的宏观时序——`update()` → `flush_effects()` → `render()` → `paint()`。本章深入帧与帧之间的底层机制：双缓冲如何工作、数据如何在帧间存活、以及为什么元素树重建而状态保留。

> 源文件参考：`src/window.rs` 的 `Frame`/`Window` 结构体、`draw()` 方法、`with_element_state()` 方法

## 双缓冲：`rendered_frame` 和 `next_frame`

`Window` 持有两个 `Frame` 实例（`src/window.rs:844-845`）：

```rust
pub struct Window {
    // ...
    pub(crate) rendered_frame: Frame,
    pub(crate) next_frame: Frame,
    // ...
}
```

每个 `Frame`（`src/window.rs:668-708`）拥有完整的帧私有数据：

```rust
pub(crate) struct Frame {
    pub(crate) element_states: FxHashMap<(GlobalElementId, TypeId), ElementStateBox>,
    pub(crate) accessed_element_states: Vec<(GlobalElementId, TypeId)>,
    pub(crate) mouse_listeners: Vec<Option<AnyMouseListener>>,
    pub(crate) dispatch_tree: DispatchTree,
    pub(crate) scene: Scene,
    pub(crate) hitboxes: Vec<Hitbox>,
    pub(crate) window_control_hitboxes: Vec<Hitbox>,
    pub(crate) deferred_draws: Vec<DeferredDraw>,
    pub(crate) input_handlers: Vec<Option<PlatformInputHandler>>,
    pub(crate) tooltip_requests: Vec<TooltipRequest>,
    pub(crate) cursor_styles: Vec<CursorStyleRequest>,
    pub(crate) focus: Option<FocusId>,
    pub(crate) window_active: bool,
    tab_stops: TabStopMap,
}
```

`rendered_frame` 保存的是**上一帧**的完整数据；`next_frame` 是**当前帧**正在构建的数据。两者互为"镜像"——每帧结束时交换。

## 一帧的完整生命周期（底层）

`Window::draw()`（`src/window.rs:1914-1943`）是交换的核心，在 `flush_effects` 中 `RefreshWindows` 效果被触发时调用：

```
Window::draw(cx)
  │
  ├── 1. draw_roots(cx)          ← 向 next_frame 中渲染新一帧
  │      ├── render()  → 构建元素树
  │      ├── request_layout() → Taffy 布局注册
  │      ├── compute_layout() → Taffy 求解
  │      ├── prepaint()
  │      │     ├── 填充 next_frame.hitboxes  ← Window::insert_hitbox()
  │      │     ├── 填充 next_frame.mouse_listeners
  │      │     ├── 填充 next_frame.cursor_styles
  │      │     └── 访问 element state → push next_frame.accessed_element_states
  │      └── paint()
  │            ├── 填充 next_frame.scene  ← Window::paint_quad/text/shadow/...
  │            └── 构建 next_frame.dispatch_tree  ← push_node/set_key_context/on_action
  │
  ├── 2. layout_engine.clear()    ← 清空 Taffy（布局结果已在 prepaint 消费完毕）
  │
  ├── 3. text_system().finish_frame()  ← 字体系统收尾
  │
  ├── 4. next_frame.finish(&mut rendered_frame)  ← ★ 状态跨帧迁移
  │
  ├── 5. mem::swap(&mut rendered_frame, &mut next_frame)  ← ★ 交换缓冲区
  │      ├── 现在的 rendered_frame = 刚画完的帧
  │      └── 现在的 next_frame = 上一帧的旧数据
  │
  └── 6. next_frame.clear()       ← 清空旧帧，准备下一轮
         ├── element_states.clear()
         ├── scene.clear()
         ├── dispatch_tree.clear()
         ├── hitboxes.clear()
         ├── mouse_listeners.clear()
         ├── input_handlers.clear()
         ├── cursor_styles.clear()
         └── ...
```

## Frame::finish() —— 状态跨帧迁移

这是整个跨帧持久化机制的核心（`src/window.rs:805-815`）：

```rust
pub(crate) fn finish(&mut self, prev_frame: &mut Self) {
    // self = next_frame（新帧，刚渲染的）
    // prev_frame = rendered_frame（上一帧的）

    for element_state_key in &self.accessed_element_states {
        // key = (GlobalElementId, TypeId)，如 ([root, list, scroll], TypeId(ScrollState))
        if let Some((element_state_key, element_state)) =
            prev_frame.element_states.remove_entry(element_state_key)
        {
            // 旧帧中有此状态 → 移动到新帧
            self.element_states.insert(element_state_key, element_state);
        }
        // 旧帧中没有 → 本轮首次创建（如新元素首次渲染、第一帧）
    }

    self.scene.finish(); // Scene 内部的排序和批处理
}
```

逻辑要点：

- **只迁移"被访问过"的状态**：`accessed_element_states` 列表由 `with_element_state()` 填充。未被访问的状态留在旧帧中，`clear()` 时丢弃。
- **移动语义**：`remove_entry` 从旧帧中取走状态，插入新帧。不是 clone——大状态的迁移开销为零（指针移动）。
- **Scene::finish()**：独立于 element state，按 `DrawOrder` 排序各图元 Vec，完成精灵的纹理批处理分组。

## with_element_state() —— 元素如何访问跨帧状态

```rust
// src/window.rs:2628-2693（简化）
pub fn with_element_state<S, R>(
    &mut self,
    global_id: GlobalElementId,
    f: impl FnOnce(Option<&mut S>, &mut Window) -> R,
) -> R {
    let key = (GlobalElementId(global_id.0.clone()), TypeId::of::<S>());

    // ★ 第一步：记录"此状态本帧被访问过了"
    // 这决定了 finish() 中该状态能否从旧帧迁移过来
    self.next_frame.accessed_element_states.push(key.clone());

    // ★ 第二步：先查 next_frame（本帧已创建过），
    //          再查 rendered_frame（上一帧的状态）
    let state_box = self.next_frame.element_states
        .remove(&key)
        .or_else(|| self.rendered_frame.element_states.remove(&key));

    // 第三步：执行用户闭包
    let (result, state) = match state_box {
        Some(mut state_box) => {
            // 旧状态存在 → 传入 &mut S
            let state: &mut S = state_box.inner.downcast_mut().unwrap();
            let result = f(Some(state), self);
            (result, ElementStateBox { inner: state_box.inner, ... })
        }
        None => {
            // 首次创建 → 传入 None
            f(None, self)
            // 如果闭包在 None 分支需要创建状态，由闭包通过其它方式存储
        }
    };

    // 第四步：存回 next_frame
    self.next_frame.element_states.insert(key, state_box);

    result
}
```

## 完整数据流示例

```
帧 N：
  rendered_frame.element_states = {
    (scroll_id, TypeId(ScrollState)) → ScrollState { offset: 150 }
  }

  Window::draw(cx) 触发：
    │
    │  paint() 阶段：元素调用 with_element_state::<ScrollState>(scroll_id, |state, window| {
    │      state.offset += 10;  // 150 → 160
    │  })
    │    → next_frame.accessed_element_states.push((scroll_id, TypeId(ScrollState)))  ← 标记
    │    → 在 rendered_frame 中找到 ScrollState { offset: 150 }                       ← 取出
    │    → 闭包修改为 ScrollState { offset: 160 }
    │    → 存入 next_frame.element_states                                             ← 存回
    │
    ├── finish(&mut rendered_frame):
    │    for key in accessed_element_states:  # key = (scroll_id, TypeId(ScrollState))
    │        remove_entry(key) from rendered_frame  → 插入 next_frame
    │    rendered_frame 中其他未被访问的状态 → 被遗弃（下一帧 clear()）
    │
    ├── mem::swap → rendered_frame 现在是刚渲染的帧
    │    rendered_frame.element_states = { (scroll_id, TypeId(ScrollState)) → ScrollState { offset: 160 } }
    │
    └── next_frame.clear() → next_frame 变为空白

帧 N+1：
  再次访问 (scroll_id, TypeId(ScrollState))
    → rendered_frame 中仍存在 ScrollState { offset: 160 } → 跨帧存活成功 ✓
```

## 什么数据跨帧存活，什么不存活

```
跨帧存活（通过 Frame::finish 迁移）：
  ✅ element_states — key 匹配的旧状态移动到新帧
  ✅ focus — 存储于 Frame.focus，在 clear() 之前通过 draw() 的 focus_path 逻辑同步

每帧从零构建（不迁移）：
  ❌ Scene 图元列表 — paint() 重新生成
  ❌ DispatchTree — paint() 阶段重新 push_node / set_key_context / on_action
  ❌ Hitboxes — prepaint() 阶段重新 insert_hitbox()
  ❌ Mouse listeners — prepaint() 重新注册
  ❌ 元素树本身 — Arena 分配，帧末批量释放
```

## 为什么元素树每帧重建但状态保留？

这是 GPUI 混合模式（hybrid immediate + retained）的本质：

| | 立即模式（Immediate） | 保留模式（Retained） |
|---|---|---|
| **范围** | 元素树、Scene、Hitboxes、DispatchTree | Element states、Focus |
| **每帧行为** | 从 `render()` 完全重建 | 通过双缓冲 key 匹配迁移 |
| **分配方式** | Arena（帧末批量释放） | HashMap（move 到新帧） |
| **优势** | 无需 diff 算法，渲染逻辑简洁 | 需要的少量状态精确持久化 |

这种设计的代价是：**状态只有在通过 `with_element_state` 显式声明"我需要跨帧存活"时才会被保留**。如果你在 `render()` 中声明了一个局部变量——它不会自动持久化。这迫使你显式区分"UI 装饰状态"和"数据状态"，与第 1 章"三大寄存器"的设计哲学一脉相承。

## 双缓冲与并发

`rendered_frame` 是"完成态"——平台渲染器从中读取 `Scene` 提交 GPU。`next_frame` 是"构建态"——当前帧的所有写入操作都在其上。交换是原子的 `mem::swap`（指针交换，不是数据拷贝），因此：

- 平台渲染器可以持续读取 `rendered_frame.scene` 而不会看到半成品。
- 新帧的构建在独立的 `next_frame` 上进行，不会与渲染器的读取冲突。
- `mem::swap` 之后，旧 `rendered_frame`（现在变成了 `next_frame`）立即被 `clear()`，释放不再需要的数据。

---

**如果你只记住一件事：** `rendered_frame` 和 `next_frame` 互为镜像。渲染写入 `next_frame`，`finish()` 从旧帧迁移被访问的状态，`mem::swap` 交换角色，`clear()` 清空旧数据。跨帧存活的数据通过 `accessed_element_states` 列表和 key 匹配精确控制，而非自动保留一切。
