# 第 2 章：帧循环解剖

GPUI 是一个"帧驱动"的框架。理解一个帧内发生了什么，你才能真正理解 `notify()` 为什么不是立即重绘，以及代码的执行顺序为什么有时令你意外。

> 源文件参考：`src/app.rs` 的 `update()` / `flush_effects()`、`src/window.rs` 的 `draw()`

## 一帧的全貌

```
App::update(callback)
  │
  ├─ 1. start_update()
  │     └─ 递增 generation 计数
  │
  ├─ 2. 执行你的 callback（事件处理 / 渲染 / 响应）
  │     └─ 你调用 cx.notify() → 推入 pending_effects: [RefreshWindows]
  │     └─ 你调用 cx.emit(E) → 推入 pending_effects: [Emit(E, ...)]
  │     └─ 你调用 cx.defer(f) → 推入 pending_effects: [Defer(f)]
  │
  ├─ 3. finish_update()
  │     └─ flush_effects()
  │           │
  │           └─ while pending_effects 非空:
  │                 ├─ Notify → 调用 observers
  │                 ├─ Emit → 调用 subscribers
  │                 ├─ RefreshWindows:
  │                 │     ├─ Window::draw()
  │                 │     │     ├─ render()          // 从根视图生成元素树
  │                 │     │     ├─ request_layout()  // 向 Taffy 注册布局
  │                 │     │     ├─ compute_layout()  // 求解布局约束
  │                 │     │     ├─ prepaint()        // 注册命中测试区域
  │                 │     │     └─ paint()           // 构建 Scene（绘制命令列表）
  │                 │     └─ 提交 Scene 到 GPU
  │                 ├─ Defer(f) → 执行延迟闭包
  │                 └─ EntityCreated → 触发实体创建通知
  │
  └─ 4. 帧结束，Arena 清理
```

## 关键洞察

### notify() 不立即重绘

```rust
cx.notify();           // 只推入一个 RefreshWindows 到队列
println!("notified!"); // 这行在重绘之前执行
// flush_effects() 在 callback 返回后才运行
```

单帧只渲染一次，无论你 `notify()` 了多少次。因为 `flush_effects()` 中的 `while` 循环会不断处理直到队列为空，但 `RefreshWindows` 效果内部会做去重。

### 效果可以产生新效果

```
notify → observer 中又 notify → 又 push 新效果 → flush 继续处理
```

这就是为什么 `flush_effects()` 用 `while` 而不是 `if`。但也正因为这种递归可能性，GPUI 内部有限制：不能在 observer 中 notify 同一个实体（会 panic）。

### render() 发生在所有 notify 处理完之后

```
你的回调 → notify 进队列 → 回调继续 → 又 notify → 回调返回
  → flush_effects: 处理所有 notify/emit
  → 最后才 RefreshWindows → render()
```

这意味着：你在一个帧中可以放心地多次修改实体状态，render 只会看到最终结果。

### 同一帧内的多次 paint

虽然一棵元素树一帧只 paint 一次，但一个窗口可能有**多层**（layers）。典型的例子是弹出层（anchored overlay）——每个 layer 会触发独立的 paint 调用。不过对于初学者，理解"一个窗口一帧一次 render"就够用了。

## 效果详解

`Effect` 是一个内部枚举（`src/app.rs`）：

```rust
enum Effect {
    Notify { emitter: EntityId },
    Emit { emitter: EntityId, event: AnyEvent },
    RefreshWindows { entity_ids: Vec<EntityId> },
    Defer { callback: Box<dyn FnOnce(&mut App) + 'static> },
    EntityCreated { entity_id: EntityId, entity_type: TypeId },
    // ...
}
```

| 效果 | 来源 | 作用 |
|------|------|------|
| `Notify` | `cx.notify()` | 触发该实体的 observers |
| `Emit` | `cx.emit(event)` | 触发该实体的 event subscribers |
| `RefreshWindows` | notify/emit 间接产生 | 标记窗口待刷新，下一轮 draw |
| `Defer` | `cx.defer(f)` | 在当前效果循环结束时执行 |
| `EntityCreated` | `cx.new()` | 通知系统新实体已创建 |

## 平台驱动的帧调度

帧循环不是 GPUI 发明的。它由操作系统的事件循环驱动：

- **macOS**：`CFRunLoop` → `requestAnimationFrame` 回调
- **Windows**：消息循环 `GetMessage` / `PeekMessage` → `WM_PAINT`
- **Linux**：`calloop` 事件循环 → Wayland `frame` 回调 / X11 `Expose`

GPUI 的 `Platform` trait 中有一个 `run()` 方法——这就是连接平台事件循环和 `App::update()` 的桥梁。平台负责：
1. 收集输入事件（键盘、鼠标）
2. 包装为 GPUI 事件
3. 调用 `App::update()` 处理
4. 请求下一帧（vsync）

## 为什么理解帧循环很重要？

1. **调试顺序问题**：如果你在 `notify()` 后立即读某个值，读到的可能是旧值——因为 effects 还没处理。
2. **性能优化**：知道 `notify()` 是廉价的（只 push 队列），你可以放心使用。真正耗时的是 `render()` 和 `paint()`。
3. **避免死循环**：observer 中再 notify 可能产生无限循环。理解 `flush_effects` 的 while 循环就能预判这种风险。

---

**如果你只记住一件事：** 你的代码在一个 `update()` 回调中运行；所有 notify/emit 之类副作用都推迟到回调返回后的 `flush_effects()` 中处理；render 在所有副作用处理完后才发生。
