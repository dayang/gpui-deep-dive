# 第 14 章：事件分发树

`DispatchTree` 是 GPUI 键盘事件分发的核心数据结构。它构建于 `paint` 阶段，消费于按键分发阶段。它不是元素树、不是 hitbox 列表——它就是一棵**真正的树**，专为键盘动作分发设计。

> 源文件参考：`src/key_dispatch.rs`（完整阅读）、`src/window.rs` 的 dispatch 相关方法

## DispatchTree 是什么

```rust
// src/key_dispatch.rs:71-80
pub(crate) struct DispatchTree {
    node_stack: Vec<DispatchNodeId>,
    pub(crate) context_stack: Vec<KeyContext>,
    view_stack: Vec<EntityId>,
    nodes: Vec<DispatchNode>,
    focusable_node_ids: FxHashMap<FocusId, DispatchNodeId>,
    view_node_ids: FxHashMap<EntityId, DispatchNodeId>,
    keymap: Rc<RefCell<Keymap>>,
    action_registry: Rc<ActionRegistry>,
}
```

- **`nodes`**：所有节点的平面存储（Vec），节点通过 `DispatchNodeId(usize)` 索引。
- **`node_stack`**：当前活跃路径（根到当前节点），维护构建时的父子关系。
- **`context_stack` / `view_stack`**：与 node_stack 同步的 key context 和 view ID 栈。
- **`focusable_node_ids` / `view_node_ids`**：反向索引——从 `FocusId` 或 `EntityId` 快速查找对应节点。

### DispatchNode

```rust
// src/key_dispatch.rs:82-91
#[derive(Default)]
pub(crate) struct DispatchNode {
    pub key_listeners: Vec<KeyListener>,
    pub action_listeners: Vec<DispatchActionListener>,
    pub modifiers_changed_listeners: Vec<ModifiersChangedListener>,
    pub context: Option<KeyContext>,
    pub focus_id: Option<FocusId>,
    view_id: Option<EntityId>,
    parent: Option<DispatchNodeId>,
}
```

注意：**节点不存储 bounds、不存储 depth、不存储 z_index**。它只存储与事件分发相关的信息：谁要监听按键、谁要监听动作、当前 key context 是什么、焦点和视图归属。命中测试和鼠标事件走的是另一套机制（`hitboxes` 列表，见下文）。

### DispatchActionListener

```rust
// src/key_dispatch.rs:132-136
pub(crate) struct DispatchActionListener {
    pub(crate) action_type: TypeId,
    pub(crate) listener: Rc<dyn Fn(&dyn Any, DispatchPhase, &mut Window, &mut App)>,
}
```

每个 action listener 存储了它关心的动作类型（`TypeId`）和一个类型擦除的回调。回调参数是 `(&dyn Any, DispatchPhase, &mut Window, &mut App)`——不是 `(&ActionType, ...)`，抹除了具体动作类型。

## 构建过程

DispatchTree 在 `paint` 阶段同步构建（`src/window.rs:3321-3341` 等）：

1. **`push_node()`**——进入一个元素时调用。创建新节点，设置 parent 为当前栈顶，推入 `node_stack`。
2. **`set_focus_id(focus_id)`**——元素设置了 focus_handle 时调用。设置当前节点的 `focus_id`，插入 `focusable_node_ids` 索引。
3. **`set_view_id(entity_id)`**——元素的 view owner 改变时调用。设置 `view_id`，插入 `view_node_ids` 索引。
4. **`set_key_context(context)`**——元素调用 `window.set_key_context()` 时，设置当前节点的 key context。
5. **`on_action(action_type, listener)`**——元素调用 `window.on_action()` 时，注册 action listener 到当前节点。
6. **`on_key_event(listener)`**——注册原始按键监听器。
7. **`pop_node()`**——退出元素时调用。弹出栈。

构建的结果是一棵树，通过 `DispatchNode.parent` 链接。

## 命中测试——与 DispatchTree 无关

鼠标事件的命中测试使用 `Frame::hit_test()`（`src/window.rs:775-797`），**不走 DispatchTree**：

```rust
pub(crate) fn hit_test(&self, position: Point<Pixels>) -> HitTest {
    let mut hit_test = HitTest::default();
    for hitbox in self.hitboxes.iter().rev() {
        let bounds = hitbox.bounds.intersect(&hitbox.content_mask.bounds);
        if bounds.contains(&position) {
            hit_test.ids.push(hitbox.id);
            if hitbox.behavior == HitboxBehavior::BlockMouse {
                break;
            }
        }
    }
    hit_test
}
```

hitboxes 是一个平面列表，按插入顺序反序遍历（后插入的先命中，对应 Z-order 高的元素）。返回的 `HitTest` 包含命中的 `HitboxId` 列表。

鼠标事件的分发通过 `Frame::mouse_listeners` ——一个与 hitboxes 平行的列表，而非 DispatchTree。

## DispatchTree 的查询能力

### dispatch_path

```rust
// src/key_dispatch.rs:550-558
pub fn dispatch_path(&self, target: DispatchNodeId) -> SmallVec<[DispatchNodeId; 32]> {
    let mut dispatch_path = SmallVec::new();
    let mut current_node_id = Some(target);
    while let Some(node_id) = current_node_id {
        dispatch_path.push(node_id);
        current_node_id = self.nodes[node_id.0].parent;
    }
    dispatch_path.reverse(); // 根 → 目标
    dispatch_path
}
```

从目标节点沿 parent 链回溯到根，然后反转——得到从根到目标的路径。这就是键盘动作分发的 Capture（根→目标）和 Bubble（目标→根）的路径。

### focus_path 和 view_path

```rust
// src/key_dispatch.rs:561-586
pub fn focus_path(&self, focus_id: FocusId) -> SmallVec<[FocusId; 8]> { ... }
pub fn view_path(&self, view_id: EntityId) -> SmallVec<[EntityId; 8]> { ... }
```

查询某个 `FocusId` 或 `EntityId` 在树中的祖先链。

## 键盘分发流程

完整流程（`src/window.rs:3843-3884`）：

```
按键事件到达
  │
  ├─ 1. 确定 dispatch_path
  │     focus.is_some() → dispatch_path(focusable_node_id)
  │     focus.is_none() → [root_node_id]
  │
  ├─ 2. dispatch_tree.dispatch_key(pending, keystroke, &dispatch_path)
  │     └─ 查询 keymap：bindings_for_input(&input, &context_stack)
  │         ├─ 部分匹配（pending=true）→ 缓存 pending keystrokes，等待下一个键
  │         ├─ 完全匹配 → 返回 bindings
  │         └─ 不匹配但有 pending → replay_prefix 重放之前的输入
  │
  ├─ 3. 对每个 binding：
  │     ├─ Capture 阶段（dispatch_path 正向）
  │     │     每个节点的 key_listeners 和 action_listeners 被调用
  │     │     DispatchPhase::Capture
  │     │
  │     └─ Bubble 阶段（dispatch_path 反向）
  │           每个节点同样被调用
  │           DispatchPhase::Bubble
  │           cx.stop_propagation() 可中止
  │
  └─ 4. 全局按键观察器（observe_keystrokes / intercept_keystrokes）
```

## 与书中之前描述的差异

本书早期版本将 `DispatchTree` 描述为"命中测试结果的平面列表（不是树）"。这是错误的。`DispatchTree` **确实是一棵树**——有 parent 指针、有节点栈、有根到叶的 dispatch path。只是节点以 Vec 存储（arena-style），而非指针图。

平面列表是 `hitboxes`（用于鼠标命中测试），它与 `DispatchTree` 是两个完全独立的系统。

## 关键总结

- `DispatchTree` 是**树**（不是平面列表），专为键盘动作分发设计。
- `DispatchNode` 存储 **监听器列表 + key context + focus/view ID**，不存储 bounds。
- 鼠标命中测试走 `hitboxes` 列表（独立的平面列表），不走 DispatchTree。
- 按键分发沿 `dispatch_path`（根→目标的路径）执行 Capture 和 Bubble 阶段。
- `DispatchPhase` 枚举：`Capture`（根→目标正向）和 `Bubble`（目标→根反向）。

---

**如果你只记住一件事：** `DispatchTree` 是为键盘事件分发构建的**树**，`DispatchNode` 存储动作/按键监听器和 key context。鼠标事件的命中测试走的是另一个独立系统（`hitboxes` 平面列表），不要混为一谈。
