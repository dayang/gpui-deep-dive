# 第 14 章：事件分发树

`DispatchTree` 是连接"元素有 hitbox"和"点击事件到达正确元素"的桥梁。它不是元素树，而是基于命中测试结果构建的、按 Z-order 排序的事件传播结构。

> 源文件参考：`src/window.rs` 的 `DispatchTree` 相关实现、`src/bounds_tree.rs`

## DispatchTree 是什么

```rust
// src/window.rs（简化）
struct DispatchTree {
    nodes: Vec<DispatchNode>,
    stack: Vec<usize>,  // 当前遍历路径
}

struct DispatchNode {
    entity_id: Option<EntityId>,
    focus_handle: Option<WeakFocusHandle>,
    depth: u32,
    z_index: u16,
    bounds: Bounds<Pixels>,
}
```

它是一个**按深度和 Z-order 排序的节点列表**，不是树形结构（尽管名字叫"tree"）。历史原因——早期版本可能是树，后来优化成了平面数组。

## 构建过程

在 `prepaint()` 阶段，当元素调用 `window.insert_hitbox(bounds, hit_test)` 时：

```
1. 创建一个 DispatchNode
2. 记录当前元素所属的 EntityId（如果有焦点的视图）
3. 记录它的 bounds（用于后续命中测试）
4. 记录它的 z_index 和 depth
5. 推入 nodes 列表
```

关键：**并非所有元素都产生 dispatch node**。只有那些在 `prepaint()` 中调用了 `insert_hitbox` 的元素——即交互式元素（设置了 `on_click`、`on_drag` 等）。

## 命中测试

当鼠标事件到达时：

```rust
// Window 中
fn hit_test(&self, position: Point<Pixels>) -> Vec<usize> {
    // 1. 用 BoundsTree 快速查找 position 命中了哪些 bounds
    // 2. 返回从高 z-index 到低 z-index 的匹配节点索引列表
    let hits = self.bounds_tree.query(position);
    // 3. 按 z_index 降序排序（顶部的元素优先接收事件）
    hits.sort_by_key(|i| Reverse(self.nodes[*i].z_index));
    hits
}
```

`BoundsTree`（`src/bounds_tree.rs`）是一个空间索引（类似 R-tree），用 O(log n) 复杂度定位命中区域。

## 事件分发：Capture + Bubble

事件分发模仿了 Web 的 DOM 事件模型：

```
假设元素树：
  Root (z=0)
    └── Panel (z=0)
          ├── Button A (z=1)   ← 鼠标点击了这里
          └── Button B (z=1)

DispatchTree 中的路径（按深度）：
  Root → Panel → Button A

Capture 阶段（从根到最深命中节点）：
  Root.on_mouse_down(capture) → Panel.on_mouse_down(capture) → Button A.on_mouse_down(capture)

Bubble 阶段（从最深命中节点到根）：
  Button A.on_mouse_down(bubble) → Panel.on_mouse_down(bubble) → Root.on_mouse_down(bubble)
```

### 代码中的体现

```rust
// Window 中处理鼠标事件
fn dispatch_mouse_event(&mut self, event: MouseDownEvent, cx: &mut App) {
    let hits = self.hit_test(event.position);

    // Capture 阶段：从顶部（根）到底部（最深的命中元素）
    for &node_idx in hits.iter() {
        if let Some(entity_id) = self.dispatch_tree.nodes[node_idx].entity_id {
            cx.update_entity(entity_id, move |view, cx| {
                view.on_mouse_event(&event, DispatchPhase::Capture, cx);
            });
            if event.propagation_stopped { return; }
        }
    }

    // Bubble 阶段：从底部到顶部
    for &node_idx in hits.iter().rev() {
        if let Some(entity_id) = self.dispatch_tree.nodes[node_idx].entity_id {
            cx.update_entity(entity_id, move |view, cx| {
                view.on_mouse_event(&event, DispatchPhase::Bubble, cx);
            });
            if event.propagation_stopped { return; }
        }
    }
}
```

### 停止传播

```rust
div().on_click(cx.listener(|this, event: &ClickEvent, cx| {
    // 处理点击
    cx.stop_propagation(); // ← 阻止事件继续传播
}))
```

`cx.stop_propagation()` 设置内部标志，导致后续的 capture/bubble 节点被跳过。

## 与键盘事件的交互

键盘事件不经过命中测试——它们发送给焦点元素（`FocusHandle`）。但 dispatch tree 仍然参与：

```
键盘事件 → 查找 focused 元素
  → 在 DispatchTree 中定位该元素的路径
  → 沿路径分发（Capture + Bubble）
  → 类似鼠标事件，但起点是焦点元素而非命中测试结果
```

## Global 事件处理器

除了元素级别的处理，GPUI 还支持全局处理器：

```rust
// 在任何元素上注册（在 App 级别）
cx.on_app_action(|cx, action: &Quit, _phase| {
    cx.quit();
});

// 全局按键观察器
cx.on_keystroke(|cx, keystroke: &KeystrokeEvent| {
    // 记录所有按键（用于调试/宏录制等）
});
```

全局处理器的优先级：
1. **Keystroke 拦截器**（`on_keystroke`）——在 dispatch 之前，可以取消。
2. **元素 dispatch**（Capture + Bubble）。
3. **全局 action 处理器**（`on_app_action`）——只在 Bubble 阶段尾部触发。

---

**如果你只记住一件事：** DispatchTree 是命中测试结果的平面列表（不是树）。事件分发分为 Capture 阶段（根→目标）和 Bubble 阶段（目标→根），任一阶段都可以 `stop_propagation()`。
